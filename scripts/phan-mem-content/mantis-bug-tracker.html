Bạn vào trang bug tracker của studio game indie, thấy hàng chục lỗi: nút nhảy không phản hồi, texture lệch, crash khi mở shop — mỗi mục có người báo, ưu tiên, trạng thái «Assigned / Resolved / Closed». Nhiều team VN dùng Mantis Bug Tracker (MantisBT): công cụ web mã nguồn mở, cài server riêng, không phụ thuộc SaaS trả phí hàng tháng.

MantisBT không phải IDE hay công cụ viết code — nó là hệ thống theo dõi bug và yêu cầu phần mềm giúp nhóm dev, QA và PM cùng nhìn một nguồn sự thật về lỗi còn tồn, ai đang xử lý và bản build nào đã sửa. Hiểu Mantis (hoặc công cụ bug tracker tương đương) giúp học sinh THPT làm dự án nhóm CS, game jam và chuẩn bị cho internship IT nơi «ghi bug rõ ràng» thường quan trọng không kém «viết code chạy được».

01Mantis Bug Tracker là gì?

Mantis Bug Tracker (viết tắt MantisBT) là phần mềm theo dõi bug và issue trên web, phát triển bởi cộng đồng mã nguồn mở, giấy phép GPL. Ra đời từ đầu thập niên 2000, MantisBT được viết bằng PHP, chạy trên server LAMP/LEMP (Linux, Apache/Nginx, MySQL/MariaDB, PHP) — bạn truy cập qua trình duyệt, không cần cài app riêng trên từng máy dev.

Khác với ghi chú bug rải rác trên Zalo, Google Sheet hay comment trong code, MantisBT gom mỗi lỗi thành một issue có ID cố định (#0001234): tiêu đề, mô tả tái hiện (steps to reproduce), mức ưu tiên, severity, người được gán, phiên bản phần mềm và timeline thay đổi trạng thái. Giao diện ban đầu trông «cổ điển» hơn Jira hay Linear, nhưng workflow rõ ràng và phù hợp team cần kiểm soát dữ liệu trên server nội bộ.

  • Issue tracking — tạo, gán, cập nhật trạng thái bug và feature request theo workflow tùy chỉnh
  • Phân quyền & dự án — nhiều project trong một instance; role Reporter, Developer, Manager
  • Thông báo email — tự động gửi mail khi issue được gán, comment hoặc đổi trạng thái
  • Báo cáo & plugin — biểu đồ, filter, Graphviz; mở rộng qua plugin cộng đồng

02Tính năng cốt lõi

1. Tạo issue, mô tả bug & tái hiện lỗi

Đơn vị trung tâm là issue (bug report hoặc feature request). Người báo điền summary ngắn («Crash khi đăng nhập bằng Google»), mô tả chi tiết, steps to reproduce (bước 1 mở app → bước 2 bấm Login → bước 3 chọn Google → crash), kèm expected vs actual result. Đính kèm screenshot, log file hoặc video ngắn — thông tin càng cụ thể, dev càng ít thời gian đoán mò.

Học sinh THPT mới làm QA cần thói quen viết bug report chuẩn: trả lời «làm sao tái hiện?», «OS nào?», «build bao nhiêu?». MantisBT có custom field thêm Device, OS version tùy dự án.

  • Category & severity — phân loại UI, gameplay, backend; mức Blocker / Critical / Major / Minor
  • Reproducibility — Always / Sometimes / Random giúp ưu tiên xử lý
  • Attachment — screenshot, crash dump, video tái hiện lỗi

2. Workflow, gán người xử lý & thông báo nhóm

Mỗi issue đi qua workflow trạng thái — ví dụ New → Acknowledged → Assigned → Resolved → Verified → Closed (team có thể tùy chỉnh). PM hoặc lead dev assign issue cho thành viên; Mantis gửi email notification khi có thay đổi — dev không cần refresh trang liên tục. Comment trong issue giữ toàn bộ trao đổi («Em thử trên build 1.2 chưa còn lỗi», «Cần thêm log từ máy Samsung») thay vì thread chat bị chìm.

Studio outsource hay cho khách nước ngoài account Reporter — mở bug trực tiếp, team VN fix và cập nhật status. CLB lập trình THPT có thể self-host một instance phục vụ 2–3 dự án song song.

3. Filter, báo cáo & quản lý nhiều dự án

MantisBT hỗ trợ nhiều project trong một cài đặt: «Website trường», «Game jam 2025», «Plugin nội bộ» — mỗi project có category, version và quyền truy cập riêng. Filter lưu truy vấn hay dùng («Open bugs assigned to me», «Critical chưa resolve tuần này»); tab Summary và biểu đồ giúp PM báo cáo tiến độ trước sprint review hoặc buổi chấm đồ án.

Luồng điển hình: tester chạy checklist beta → mở issue → standup lọc «Assigned» → dev resolve → regression → close. Tích hợp API/CI giúp liên kết với IDE — nắm quy trình này là kỹ năng giao hàng phần mềm thực tế.

03So sánh với phần mềm tương tự

Hai lựa chọn hay được đặt cạnh MantisBT nhất là Jira (Atlassian, SaaS/on-prem, phổ biến enterprise) và GitHub Issues (tích hợp sẵn repo Git, miễn phí cho dự án open source). Không có câu trả lời «đúng duy nhất» — tùy quy mô team, ngân sách, yêu cầu self-host và mức độ gắn với Git workflow.

MantisBT vs Jira

Jira là «chuẩn công nghiệp» ở nhiều công ty lớn: Scrum board, Epic, Sprint, tích hợp Confluence và marketplace app khổng lồ — nhưng chi phí license và độ phức tạp cấu hình cao. MantisBT miễn phí, mã nguồn mở, cài trên server riêng, giao diện gọn hơn, tập trung vào bug tracking thuần — phù hợp SME Việt Nam, team outsource và trường học muốn kiểm soát dữ liệu không gửi lên cloud nước ngoài.

Điểm mạnh và yếu so sánh

MantisBT mạnh hơn: chi phí bằng không (self-host), cài đặt nhanh cho team nhỏ, workflow bug đơn giản, phù hợp QA thuần và khách hàng chỉ cần portal báo lỗi.
Jira mạnh hơn: Agile/Scrum đầy đủ, roadmap, tích hợp enterprise (SSO, audit), plugin và báo cáo nâng cao cho hàng trăm dev.
Chọn MantisBT khi: ngân sách hạn chế, cần host nội bộ, team 5–30 người tập trung fix bug hơn quản lý product phức tạp.
Chọn Jira khi: công ty đã dùng Atlassian stack, cần Scrum scale lớn và budget cho license.

MantisBT vs GitHub Issues

GitHub Issues gắn repository Git: link PR «Fixes #42», label và milestone cùng nền tảng code — lý tưởng open source. MantisBT tách bug tracking khỏi repo: tester/khách không cần quyền Git; nhiều công ty dùng cả hai — Mantis cho QA, tham chiếu sang GitHub cho dev.

Chọn GitHub Issues khi: dự án học sinh/indie trên GitHub, mọi người đều dev và muốn issue + code một chỗ.

04Học Mantis Bug Tracker như thế nào?

1
Hiểu vai trò bug tracker trong dự án phần mềm

Trước khi cài Mantis, hãy nắm vì sao cần hệ thống issue: dev cần mô tả lỗi tái hiện được, PM cần biết còn bao nhiêu bug Critical, tester cần verify fix trên đúng build. Đọc 2–3 bug report mẫu tốt trên GitHub hoặc Mantis demo — so sánh với tin nhắn «app bị lỗi rồi» không chi tiết. Mục tiêu tuần đầu: viết được một bug report giả định cho game Unity hoặc web đồ án lớp 12 của bạn.

2
Dùng bản demo hoặc cài MantisBT local

Truy cập trang demo chính thức MantisBT hoặc cài trên XAMPP/Docker local (PHP + MySQL) — không cần server production ngay. Tạo account Reporter và Developer thử nghiệm, tạo project «Test THPT», điền vài issue giả. Làm quen menu View Issues, Report Issue, Change Status — giao diện không hiện đại như Notion nhưng logic rõ: mỗi thao tác đều ghi vào lịch sử issue.

3
Thực hành workflow và viết bug chuẩn

Trong nhóm 3–4 bạn, một người đóng vai tester: chơi thử bản game/web và mở ít nhất 5 issue với steps to reproduce, severity và screenshot. Dev assign cho mình, comment «investigating», resolve khi sửa xong — tester verify và close. Luyện thêm custom field (OS, browser, build number). Sau 1–2 tuần bạn sẽ thấy khác biệt rõ so với track bug bằng chat nhóm.

4
Filter, báo cáo và liên kết với Git

Tạo filter «My open bugs» và «All unresolved Critical»; xem tab Summary trước buổi họp nhóm. Thử ghi trong comment issue mã commit hoặc link PR GitHub («fixed in a1b2c3d») — dù Mantis không bắt buộc Git, thói quen traceability này là chuẩn industry. Nếu dự án học sinh chỉ dùng GitHub, so sánh song song một issue trên cả hai nền tảng để hiểu trade-off.

5
Áp dụng vào dự án thật: game jam, web trường, Capstone

Chọn dự án có deadline (game jam, web CLB): rule «mọi lỗi qua Mantis/Issues, không báo miệng». Trưởng nhóm review dashboard hàng ngày — dạy quy trình giao hàng: team biết còn gì chưa xong.

6
Mở rộng: QA mindset, Jira/GitHub và hướng nghề

Tìm hiểu ISTQB cơ bản hoặc khóa Software Testing miễn phí — bug tracker chỉ là công cụ, tư duy test case mới là lõi. Ghi portfolio: «Quản lý XX issue trên MantisBT cho dự án YY». Hướng nghề: QA Engineer, Test Automation — nhiều công ty VN tuyển junior QA dựa trên dự án và bug report mẫu.
